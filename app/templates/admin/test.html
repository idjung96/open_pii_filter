PII 검사 테스트

게시글 본문/제목과 첨부파일을 입력하면 일반 API POST /v1/detect/post 를 호출하여 결과를 표시합니다. HMAC 인증, audit 미들웨어, 응답 정책 등 운영 호출과 동일한 흐름을 거치므로 API 사용 이력 / PII 탐지 이력에도 그대로 기록됩니다.

첨부파일 비동기 흐름 안내
  • 첨부가 있으면 본문은 즉시 응답 (HTTP 202 ACK-3001), 첨부는 워커가 fetch → 검사 → callback_url 로 webhook 회신 흐름을 그대로 탑니다.
  • 이 페이지는 업로드 파일을 메모리에 staging 한 뒤, 임시 fetch_url/callback_url 을 같은 서버 주소로 발급하여 운영 비동기 흐름을 그대로 재현합니다 (token 기반, TTL 10분).
  • OCR 부하 폭주 방지를 위해 이 검사 테스트에 한해 OCR 은 PDF/TIFF 의 첫 2 페이지까지만 실행합니다 (PaddleOCR 기본 엔진). 운영 트래픽은 모든 페이지를 스캔합니다.
  • callback 회신은 최대 90초까지 대기하며, 시간 초과 시 화면에 별도로 표시합니다.
입력
허용 MIME: PDF / Office (DOCX·XLSX·PPTX) / 이미지 (PNG·JPEG·TIFF·BMP·WEBP·GIF) / TXT. HWP·HWPX·ZIP 등 차단 대상 형식은 본문 검사 단계에서 BLOCK 응답을 보냅니다. 스캔 PDF·다중 페이지 TIFF 는 이 페이지에서만 첫 2페이지까지 OCR 합니다.
초기화
strictness 동작 방식

PII 분석 엔진은 각 탐지 결과마다 0.0~1.0 사이의 신뢰도 점수(score)를 산출합니다. strictness는 그 점수를 PASS / BLOCK 중 어느 등급으로 분류할지 결정하는 임계값입니다. 점수가 동일해도 strictness 설정에 따라 판정이 달라집니다. Phase 9D 이후 WARN 등급은 폐기되어 2단계로 단순화됐습니다.

strictness BLOCK 임계 (score ≥) 특징
low 0.65 오탐 ↓ / 미탐 ↑ — 게시 자유도 우선
medium 0.78 기본값 — 일반 게시판에 권장
high 0.88 오탐 ↑ / 미탐 ↓ — 민원/법무 게시판 등 보호 우선
판정 규칙
  • score ≥ BLOCK 임계 → BLOCK 게시 차단
  • score < BLOCK 임계 → PASS 통과
예시 — 동일한 탐지 (score = 0.72)가 strictness별로 다르게 분류됩니다:
  • low (0.72 ≥ 0.65) → BLOCK
  • medium (0.72 < 0.78) → PASS
  • high (0.72 < 0.88) → PASS
API 호출 시 지정 방법 
{
  "request_id": "...",
  "post": { ... },
  "options": { "strictness": "medium" }
}

미지정 시 기본값은 medium. 운영 환경에서는 게시판 성격에 맞게 클라이언트가 선택해 호출합니다. pii_policies 테이블에 (entity_type, score band) → action 행이 등록되어 있으면 strictness 기본 분류 위에 패턴별 override 가 적용됩니다 (예: 특정 entity_type 을 LOG_ONLY 로 강등하여 PASS 화).
